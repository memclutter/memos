# Skills

## Purpose

Skills are reusable, named procedures an agent invokes instead of re-deriving
steps each time. `skills/` is the canonical home; each tool gets a generated copy
(a shim) so the same skill is discoverable from Claude, Cursor, Codex, and
OpenCode.

## Behaviour

- A skill is named `{category}.{entity}.{action}` and lives in a folder of the
  same name with a `SKILL.md` (frontmatter + instructions). `sys` = managing the
  OS itself.
- The canonical `SKILL.md` carries only general frontmatter; the `x-shim` block
  declares tool-specific keys (e.g. Claude's `allowed-tools`).
- Project-scoped skills take the project as an argument and, if it's missing,
  scan `projects/` and ask the owner — they never guess.
- Shims are thin `SKILL.md` files under `.claude/`, `.cursor/`, `.codex/`, and
  `.opencode/` that point back to the canonical file. They are generated, never
  hand-edited, and regenerated by `uv run memos shimify` (see
  [cli.md](cli.md)). After regenerating, skills are reloaded in the tool.
- Current skills:
  - `sys.project.specify` — create a project and write its living spec.
  - `sys.project.release` — cut a SemVer release from Conventional Commits.
  - `sys.task.specify` — start a task; write `spec.md` as a delta.
  - `sys.task.plan` — write a task's `plan.md`.
  - `sys.task.breakdown` — slice spec + plan into a checklist.
  - `sys.task.finish` — merge the task's delta into `spec/` and close it.
  - `doc.prose.review` — audit Markdown prose against an agent-style ruleset.

## Success criteria

- Every folder in `skills/` has a matching shim in each of the four tool
  directories, in sync with the canonical file. Shim sync is verifiable with
  `uv run memos doctor` ([cli.md](cli.md)).
- No shim is hand-edited; `shimify` reproduces them exactly.
- Skill names follow the three-part shape and match their folder name.
